    Implementar:

    ```python
    from collections.abc import AsyncIterator
    from contextlib import asynccontextmanager
    from sqlalchemy import MetaData
    from sqlalchemy.ext.asyncio import (
        AsyncEngine,
        AsyncSession,
        async_sessionmaker,
        create_async_engine,
    )
    from sqlalchemy.orm import DeclarativeBase

    from app.config.settings import settings

    NAMING_CONVENTION = {
        "ix": "ix_%(column_0_label)s",
        "uq": "uq_%(table_name)s_%(column_0_name)s",
        "ck": "ck_%(table_name)s_%(constraint_name)s",
        "fk": "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
        "pk": "pk_%(table_name)s",
    }

    metadata = MetaData(naming_convention=NAMING_CONVENTION)

    class Base(DeclarativeBase):
        metadata = metadata

    def create_db_engine() -> AsyncEngine:
        return create_async_engine(
            settings.postgres.async_url,
            pool_size=10,
            max_overflow=5,
            pool_pre_ping=True,
            pool_recycle=1800,
            echo=False,
            future=True,
        )

    def create_session_factory(engine: AsyncEngine) -> async_sessionmaker[AsyncSession]:
        return async_sessionmaker(engine, expire_on_commit=False, class_=AsyncSession)

    @asynccontextmanager
    async def session_scope(
        session_factory: async_sessionmaker[AsyncSession],
    ) -> AsyncIterator[AsyncSession]:
        async with session_factory() as session:
            try:
                yield session
                await session.commit()
            except Exception:
                await session.rollback()
                raise
    ```

    Notas:
    - `naming_convention` definida para alembic autogenerate consistente (futuras fases lo agradecen).
    - `Base` aquí declarado pero F1 no define tablas SQLAlchemy — el schema queda vacío del lado de la app (toda persistencia F2+).
    - `pool_size=10 + max_overflow=5` = 15 conexiones max desde el engine de app. Sumar a las del checkpointer (psycopg pool). Postgres en Railway tiene `max_connections=100` default, sobrado.
    - `pool_pre_ping=True` evita "broken pipe" en conexiones idle (Railway puede cerrar inactivas tras N min).
    - `async_url` viene de `PostgresSettings` (plan 02) y reemplaza el scheme a `postgresql+asyncpg://`.

    **Archivo 2: `app/config/redis.py`** — Redis async client con ConnectionPool (per RESEARCH.md Standard Stack + Pattern 1):

    ```python
    from redis.asyncio import ConnectionPool, Redis
    from app.config.settings import settings

    def create_redis_pool() -> tuple[Redis, ConnectionPool]:
        pool = ConnectionPool.from_url(
            settings.redis.redis_url.get_secret_value(),
            max_connections=20,
            decode_responses=False,  # we manage encoding per call-site (binary safe by default)
        )
        client = Redis(connection_pool=pool)
        return client, pool

    async def close_redis_pool(client: Redis, pool: ConnectionPool) -> None:
        await client.aclose()
        await pool.disconnect(inuse_connections=True)
    ```

    Notas:
    - `decode_responses=False` es default seguro. Tools que esperan str hacen `.decode()` explícito; binary-safe payloads (imágenes en F4) no se corrompen.
    - El `pool` se retorna para que el lifespan pueda cerrarlo en shutdown explícitamente (per Redis async docs).

    Implementar el wrapper del `AsyncPostgresSaver` que el lifespan (plan 04) va a usar con el patrón explícito `__aenter__`/`__aexit__` (per CONTEXT.md D-06 + RESEARCH.md Pattern 1 + Pitfall 1):

    ```python
    from typing import Any
    from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver

    from app.config.settings import settings

    def build_checkpointer_cm() -> Any:
        """Retorna el async context manager (no entrado todavía) de AsyncPostgresSaver.

        El lifespan debe llamar `await cm.__aenter__()` para obtener el saver, guardarlo
        en app.state.checkpointer, y al shutdown llamar `await cm.__aexit__(None, None, None)`.

        IMPORTANTE: el `setup()` (creación de tablas) se hace una sola vez. Para evitar
        "relation already exists" en restarts: confiar en la migración alembic 0001 que
        ejecuta setup() explícitamente. Si el deploy se hace antes de correr migraciones,
        el primer lifespan startup podrá llamar `await saver.setup()` — es idempotente.
        """
        return AsyncPostgresSaver.from_conn_string(
            settings.postgres.postgres_url.get_secret_value()
        )
    ```

    Notas:
    - El nombre `build_checkpointer_cm` deja explícito que retorna un context manager, no el saver instanciado. El lifespan se encarga de entrar/salir.
    - El conn string que recibe `AsyncPostgresSaver` es el URL Postgres SIN el `+asyncpg` (psycopg 3 entiende `postgresql://` directo). Por eso usamos `postgres_url.get_secret_value()` y no `async_url` (que es para SQLAlchemy/asyncpg).
    - Setup idempotente: AsyncPostgresSaver.setup() crea tres tablas (`checkpoints`, `checkpoint_writes`, `checkpoint_blobs`) + índices. Es seguro llamarla múltiples veces (la propia LangGraph maneja una tabla de migraciones interna). Aún así, preferimos correrlo en alembic init para que el orden de bootstrap sea deterministic.

    ```python
    import asyncio
    from logging.config import fileConfig
    from sqlalchemy import pool
    from sqlalchemy.engine import Connection
    from sqlalchemy.ext.asyncio import async_engine_from_config
    from alembic import context

    from app.config.settings import settings
    from app.config.db import metadata as target_metadata
    # NOTE: import any models module(s) here so their tables register on Base.metadata.
    # F1 has no app models yet; this is the hook future phases will use.

    config = context.config

    if config.config_file_name is not None:
        fileConfig(config.config_file_name)

    config.set_main_option("sqlalchemy.url", settings.postgres.async_url)

    def do_run_migrations(connection: Connection) -> None:
        context.configure(
            connection=connection,
            target_metadata=target_metadata,
            compare_type=True,
            compare_server_default=True,
        )
        with context.begin_transaction():
            context.run_migrations()

    async def run_async_migrations() -> None:
        connectable = async_engine_from_config(
            config.get_section(config.config_ini_section, {}),
            prefix="sqlalchemy.",
            poolclass=pool.NullPool,
        )
        async with connectable.connect() as connection:
            await connection.run_sync(do_run_migrations)
        await connectable.dispose()

    def run_migrations_online() -> None:
        asyncio.run(run_async_migrations())

    if context.is_offline_mode():
        # Offline mode unused in F1; raise to fail loud if invoked accidentally.
        raise RuntimeError("Offline alembic mode is not configured for this project.")
    else:
        run_migrations_online()
    ```

    Crear `alembic/versions/0001_initial_schema.py` manualmente (NO `alembic revision --autogenerate` aquí: queremos control explícito sobre la primera migración, y AsyncPostgresSaver.setup() no se detecta vía autogenerate):

    ```python
    """initial schema: langgraph checkpoint tables

    Revision ID: 0001
    Revises:
    Create Date: 2026-06-27 00:00:00.000000

    """
    from collections.abc import Sequence

    from alembic import op
    from langgraph.checkpoint.postgres import PostgresSaver
    from sqlalchemy import text

    revision: str = "0001"
    down_revision: str | None = None
    branch_labels: Sequence[str] | None = None
    depends_on: Sequence[str] | None = None


    def upgrade() -> None:
        # LangGraph checkpoint tables. We use the sync PostgresSaver inside the alembic
        # context (alembic runs sync migrations under run_sync); the sync variant shares
        # schema with AsyncPostgresSaver. setup() is idempotent.
        bind = op.get_bind()
        # The sync PostgresSaver expects a psycopg connection; alembic gives us a SQLAlchemy
        # connection. Easiest path: execute the DDL directly via raw SQL extracted from
        # langgraph-checkpoint-postgres source (stable across 3.x). Equivalent to setup().
        bind.execute(text("""
            CREATE TABLE IF NOT EXISTS checkpoints (
                thread_id TEXT NOT NULL,
                checkpoint_ns TEXT NOT NULL DEFAULT '',
                checkpoint_id TEXT NOT NULL,
                parent_checkpoint_id TEXT,
                type TEXT,
                checkpoint JSONB NOT NULL,
                metadata JSONB NOT NULL DEFAULT '{}',
                PRIMARY KEY (thread_id, checkpoint_ns, checkpoint_id)
            );
        """))
        bind.execute(text("""
            CREATE TABLE IF NOT EXISTS checkpoint_blobs (
                thread_id TEXT NOT NULL,
                checkpoint_ns TEXT NOT NULL DEFAULT '',
                channel TEXT NOT NULL,
                version TEXT NOT NULL,
                type TEXT NOT NULL,
                blob BYTEA,
                PRIMARY KEY (thread_id, checkpoint_ns, channel, version)
            );
        """))
        bind.execute(text("""
            CREATE TABLE IF NOT EXISTS checkpoint_writes (
                thread_id TEXT NOT NULL,
                checkpoint_ns TEXT NOT NULL DEFAULT '',
                checkpoint_id TEXT NOT NULL,
                task_id TEXT NOT NULL,
                idx INTEGER NOT NULL,
                channel TEXT NOT NULL,
                type TEXT,
                blob BYTEA NOT NULL,
                task_path TEXT NOT NULL DEFAULT '',
                PRIMARY KEY (thread_id, checkpoint_ns, checkpoint_id, task_id, idx)
            );
        """))
        bind.execute(text("""
            CREATE TABLE IF NOT EXISTS checkpoint_migrations (
                v INTEGER PRIMARY KEY
            );
        """))
        bind.execute(text("""
            INSERT INTO checkpoint_migrations (v) VALUES (1)
            ON CONFLICT DO NOTHING;
        """))


    def downgrade() -> None:
        op.execute("DROP TABLE IF EXISTS checkpoint_writes")
        op.execute("DROP TABLE IF EXISTS checkpoint_blobs")
        op.execute("DROP TABLE IF EXISTS checkpoints")
        op.execute("DROP TABLE IF EXISTS checkpoint_migrations")
    ```

    Notas críticas:
    - El esquema embebido es el de langgraph-checkpoint-postgres 3.x (verificable contra `langgraph.checkpoint.postgres.PostgresSaver.MIGRATIONS` en el código fuente del paquete instalado). Si en una verificación posterior `AsyncPostgresSaver.setup()` quisiera correr migraciones >= 2 (e.g. añade índices) lo hará al startup porque `IF NOT EXISTS` no rompe, y la tabla `checkpoint_migrations` con v=1 le indica el punto desde el cual continuar.
    - Alternativa más limpia (y preferible si el ejecutor verifica que funciona): correr `await saver.setup()` desde el lifespan ANTES de yield, con un flag idempotente. La migración alembic 0001 entonces se vuelve **una migración explícita vacía** (placeholder para que `alembic upgrade head` no diga "no migrations") y `setup()` queda en lifespan. El ejecutor debe escoger una de las dos rutas y documentar.
    - **Recomendación**: si en runtime de verificación, importar `langgraph.checkpoint.postgres.PostgresSaver.MIGRATIONS` y leer las DDL strings desde ahí — más a prueba de drift que hardcodear.

    Si la opción "DDL hardcoded" se siente frágil, sustituir el cuerpo de `upgrade()` por:

<threat_model>

## Trust Boundaries

| Boundary | Description |
|---|---|
| app → Postgres | Two distinct drivers (asyncpg + psycopg) share DB; pool exhaustion threat |
| app → Redis | Single pool; rate limit token storage in F5 |
| alembic env → settings | Migrations read DB URL from settings (no hardcoded creds in alembic.ini) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-01-10 | Denial of service | Pool exhaustion (asyncpg + psycopg both holding ~15 conns each) | mitigate | pool_size 10 + max_overflow 5 = 15 per engine; sum ≤ 30 vs Postgres default max_connections=100. Document caps in checkpointer module docstring. |
| T-01-11 | Tampering | DB credentials in alembic.ini committed | mitigate | alembic.ini has NO `sqlalchemy.url`; env.py composes from settings at runtime |
| T-01-12 | Information disclosure | Postgres connection string in logs (e.g. SQLAlchemy echo or alembic INFO) | mitigate | `echo=False` on engine; alembic logger filtered by structlog level; passwords never logged because `SecretStr.get_secret_value()` returns plain str only in the connection line (kept in process memory, not in log records) |
| T-01-13 | Tampering | Migration 0001 fails partially, leaving inconsistent state | mitigate | DDL uses `IF NOT EXISTS`; downgrade drops cleanly; idempotent so retry safe |
| T-01-14 | Tampering | langgraph-checkpoint-postgres schema drift across versions | accept | Version pinned to 3.1.0 in pyproject. If upgraded, regenerate migration explicitly. Documented in alembic/README. |

</threat_model>
