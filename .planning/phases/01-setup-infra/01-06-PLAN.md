       El template aprovisiona:
       - `chatwoot-rails` (web + Sidekiq combinados, image: `chatwoot/chatwoot:v3.x`)
       - `chatwoot-postgres` (Postgres + pgvector extension)
       - `chatwoot-redis`

       Si el template moderno separa Rails y Sidekiq en services distintos, dejar la separación.

    2. Configurar las env vars críticas del template (Railway prompt al deploy):
       - `FRONTEND_URL=https://chat.landatech.org`
       - `INSTALLATION_ENV=railway`
       - `SECRET_KEY_BASE` — generar con `openssl rand -hex 64` (Chatwoot lo usa para cookies firmadas)
       - `DEFAULT_LOCALE=es`
       - `MAILER_SENDER_EMAIL=no-reply@landatech.org` (placeholder OK si no hay SMTP aún; el wizard no requiere email funcional para crear el admin)
       - `SMTP_*` — placeholder vacío en F1; ajustar en F3/F7 cuando se quiera envío de notifications
       - `STORAGE_BUCKET_NAME=local` — usar storage en disk del container en F1; F5 puede migrar a S3

       Memoria mínima por service (RESEARCH.md Pitfall 5): chatwoot-rails ≥4GB. Si Railway Hobby plan total es 8GB compartido, asignar 4GB a chatwoot-rails y dejar 1-2GB para sidekiq. Si memoria es insuficiente, el operador debe aceptar upgrade a Pro o ajustar.

    3. Esperar a que los 4 services queden DEPLOYED (5-15min para el primer cold start de Rails).

    4. Configurar DNS de `landatech.org` (RESEARCH.md Pitfall 6 — proceso muy específico):

       a. En el dashboard del DNS provider (Cloudflare, Route53, lo que LANDA use):
          - Crear registro `CNAME chat.landatech.org` apuntando al dominio que Railway asignó al service `chatwoot-rails` (algo como `chatwoot-rails-production.up.railway.app`).
          - Crear registro `TXT _acme-challenge.chat.landatech.org` con el valor que Railway provea al añadir el custom domain.

       b. En Railway service `chatwoot-rails` → Settings → Networking → Custom Domain → añadir `chat.landatech.org`. Railway mostrará los dos registros DNS esperados; verificar coincidencia exacta con los que se pusieron en (a).

       c. **CRÍTICO** (RESEARCH.md Pitfall 6): Si Cloudflare está en frente, dejar el proxy en gray cloud (DNS only) hasta que Railway muestre "Certificate issued". Si está en orange cloud (proxied), el ACME challenge falla.

       d. Esperar 5-15 min. Si tras 20 min Railway sigue "pending", verificar:
          - Que ambos records DNS resolven: `dig chat.landatech.org CNAME +short` y `dig _acme-challenge.chat.landatech.org TXT +short`
          - Que el TXT record coincide letra por letra con el que Railway dio
          - Que no se hicieron retries rápidos (Let's Encrypt rate limit: 5 fallos por hora)

    5. Una vez "Certificate issued" en Railway, navegar a `https://chat.landatech.org`. Debería cargar el onboarding wizard de Chatwoot.

    6. Completar wizard:
       - Crear super admin con email del operador (`maxi@landatech.org` o similar) + password fuerte
       - Crear primera cuenta: nombre "DPG Seguros" o "LANDA Tech" (sobrescribible)
       - **Skipear** integraciones (WhatsApp, Slack, etc.) — eso es F3

    7. Verificar idle:
       - Dashboard Chatwoot abierto, status "Online"
       - No inboxes configurados, no agents extra, no contactos
       - `railway logs --service chatwoot-rails` muestra el server idle (request count ~0)

<threat_model>

## Trust Boundaries

| Boundary | Description |
|---|---|
| Internet → chat.landatech.org | Public web app; admin login + (future) WhatsApp webhook receiver |
| chatwoot-rails ↔ chatwoot-postgres | Private network in same Railway project |
| chat.landatech.org ↔ agent-app | F3+ via public HTTPS (or future internal proxy) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-01-27 | Spoofing | Anyone with the domain can attempt admin login | mitigate | Strong admin password (operator responsibility, vaulted); F5 will add rate limiting / IP allowlist via Cloudflare |
| T-01-28 | Tampering | Chatwoot Postgres exposed publicly | mitigate | Railway template default keeps DB internal; verify no public TCP port on chatwoot-postgres |
| T-01-29 | Information disclosure | Chatwoot logs may include phone numbers / message content | accept (F1) | Chatwoot's own log redaction is out of scope F1 — phase is idle. F3 will tag this as a concern when real traffic starts |
| T-01-30 | Tampering | Outdated Chatwoot image with known CVEs | mitigate | Template pins `chatwoot/chatwoot:v3.x` from upstream; F7 runbook adds "monthly upgrade" task |
| T-01-31 | Tampering | Default SECRET_KEY_BASE | mitigate | Use `openssl rand -hex 64` for SECRET_KEY_BASE (not the template default if any) |
| T-01-SC | Tampering | Railway community template legitimacy | mitigate | Template `chatwoot-all-in-one-pgvector` is community-maintained; verify it deploys the official upstream image `chatwoot/chatwoot:v3.x` (check service config post-deploy) |

</threat_model>
