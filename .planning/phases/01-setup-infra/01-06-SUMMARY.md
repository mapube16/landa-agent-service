### Tarea 2 (humano + CLI) — Deploy del template + fix de env vars + healthcheck

Operador (Maxi) hizo el click "Deploy" del template Railway. Las 3 services aparecieron (`Chatwoot`, `pgvector`, `Redis-4p8s`) pero el primer deploy quedó atrapado en un loop de `-p: - no response` infinito (rate-limit del log a los segundos).

**Diagnóstico**: el template definió 11+ env vars CRÍTICAS como **claves vacías** (`DATABASE_URL=`, `REDIS_URL=`, `RAILS_ENV=`, `INSTALLATION_ENV=`, etc.). Por eso el shellscript del entrypoint tiraba `redis-cli -p $PORT ping` con `$PORT` vacío → spam.

**Fix #1 (env vars)** vía `railway variable set --service Chatwoot`:
- `DATABASE_URL=${{pgvector.DATABASE_URL_PRIVATE}}` + `PGHOST_PRIVATE`/`PGPORT_PRIVATE`/etc. (los `*_PRIVATE` son los que pgvector expone realmente; los públicos quedaron vacíos en este template)
- `REDIS_URL=redis://default:<password>@redis-4p8s.railway.internal:6379` (construido manualmente porque Redis-4p8s tampoco populó `REDIS_URL`; solo expuso `REDIS_PASSWORD`)
- `RAILS_ENV=production`, `NODE_ENV=production`, `INSTALLATION_ENV=Railway`, `DEFAULT_LOCALE=es`, `ACTIVE_STORAGE_SERVICE=local`, `ENABLE_ALPINE_PRIVATE_NETWORKING=true`, `FORCE_SSL=true`

**Segundo síntoma**: redeploy → Rails booteó + Sidekiq procesó cron jobs (logs visibles), pero healthcheck `/api` retornaba 404 y mataba el container tras 5 min. Chatwoot no tiene endpoint `/api`.

**Fix #2 (healthcheck)** vía Railway GraphQL API (la CLI no expone `serviceInstanceUpdate`):
- Primero intenté `healthcheckPath=/` — falló igual porque pre-onboarding ningún path retorna 200 (la DB todavía no tiene rows). El Rails edge devolvía `x-railway-fallback: true 404` porque el container no estaba registrado.
- Después seteé `healthcheckPath=""` (disabled) → SUCCESS al primer try.

**Tercer detalle**: `railway redeploy` no recarga la config del service. Hay que disparar deploy fresco con un cambio de env var dummy (`TRIGGER_REBUILD=<timestamp>`).

## Desviaciones del plan

1. **[Template bug — Rule 3 auto-fix] Env vars críticas como claves vacías**
   - Síntoma: container loop con `-p: - no response`.
   - Causa: `chatwoot-all-in-one-pgvector` template define DATABASE_URL/REDIS_URL/RAILS_ENV/etc. con valores vacíos en lugar de referencias `${{...}}` automáticas. Probable bug del template o quirk de Railway template engine.
   - Fix: `railway variable set` con 11 vars correctas.
   - **No-commits** — solo runtime Railway config.

2. **[Template bug — Rule 3 auto-fix] Healthcheck path apuntando a `/api` (404 en Chatwoot)**
   - Síntoma: deploys FAILED tras 5 min de healthcheck retry, aunque Rails y Sidekiq booteaban OK.
   - Fix vía GraphQL API: disabled durante onboarding, re-enabled a `/` post-wizard.
   - **No-commits** — solo Railway config.

3. **[Railway CLI gap — workaround documentado] `railway redeploy` no recarga service-instance config**
   - Síntoma: cambié healthcheckPath via GraphQL, hice `railway redeploy`, deploy nuevo siguió usando el config viejo (cached).
   - Workaround: setear un env var dummy (`TRIGGER_REBUILD=<timestamp>`) para forzar fresh deploy.
   - Documentado en `CHATWOOT_NOTES.md` para futuros operadores.

4. **[Railway CLI gap — workaround documentado] CLI no expone `serviceInstanceUpdate`**
   - Síntoma: no hay `railway service update --healthcheck-path /` ni similar.
   - Workaround: usar Railway GraphQL API directamente con el accessToken local del CLI (`~/.railway/config.json`).
   - Documentado en `CHATWOOT_NOTES.md`.

5. **[Plan deviation — diferido] Custom domain `chat.landatech.org` no configurado**
   - El plan original requería SSL Let's Encrypt en `chat.landatech.org` como deliverable.
   - Operador decidió diferir todos los custom domains (Chatwoot + agent service) al final de Phase 1 para no bloquear deploy verification con DNS work.
   - **No bloquea Phase 1**: Chatwoot está accesible y wizard completado en la URL Railway-default. La migración a custom domain es un env-var change + DNS records (procedimiento documentado en CHATWOOT_NOTES).
