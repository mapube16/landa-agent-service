### Tarea 3 — Debugging del deploy + alembic stamp + smoke (commits `649ee10`, `58c06cc`, `e0738ea`, `8921197`)

#### Sub-tarea 3.1 — Dockerfile cache-mount (commits `649ee10` → `58c06cc`)

Primer build falló en línea 12 del `Dockerfile`:
```
dockerfile invalid: flag '--mount=type=cache,target=/root/.cache/uv' is missing an id argument
```

Primer fix (`649ee10`): agregar `id=uv-cache`. También aproveché para agregar el `COPY alembic ./alembic` y `COPY alembic.ini ./` que faltaban (defensive — esos archivos los necesitan los migrations de F1+).

Segundo error después del fix:
```
dockerfile invalid: flag '--mount=type=cache,id=uv-cache,target=/root/.cache/uv' is missing the cacheKey prefix from its id
```

Railway's BuildKit fork pide que el `id` tenga un prefix específico (e.g. `s/<service-id>-...`). En vez de cazar el prefix correcto, dropeé la cache mount entera (`58c06cc`): es solo speedup, no funcional. Cold build pasó de ~15 s objetivo a ~2 min real.

#### Sub-tarea 3.2 — agent-worker crash loops (commits `e0738ea`, `8921197`)

Tras el primer build exitoso, el worker crasheaba con:
```
RuntimeError: at least one function or cron_job must be registered
```

ARQ no tolera `WorkerSettings.functions=[]`. Fix `e0738ea`: agregar `async def _noop(ctx)` y `functions=[_noop]`. Placeholder que se borra en F2 cuando llegan los jobs reales.

Segundo crash:
```
redis.exceptions.ConnectionError: Error Multiple exceptions: [Errno 111] Connect call failed
('::1', 6379, 0, 0), [Errno 111] Connect call failed ('127.0.0.1', 6379)
connecting to localhost:6379.
```

El comentario del worker.py original asumía que ARQ lee `REDIS_URL` del env — **es falso**. `RedisSettings()` default es `host='localhost'`. Fix `8921197`: `redis_settings = RedisSettings.from_dsn(settings.redis.url.get_secret_value())`. Mismo `settings.redis.url` que usa el FastAPI app → single source of truth. Pre-commit mypy tuvo que agregar `arq==0.28.0` a `additional_dependencies` (continuación del patrón 01-02/01-04).

#### Sub-tarea 3.3 — `alembic stamp head` (no commit — Postgres-side)

`railway run --service landa-agent-service -- alembic upgrade head` falló en local: `socket.gaierror: getaddrinfo failed` porque `postgres.railway.internal` solo resuelve dentro de la red Railway.

`railway ssh --service landa-agent-service "alembic upgrade head"` levantó:
```
RuntimeError: asyncio.run() cannot be called from a running event loop
sys:1: RuntimeWarning: coroutine '_apply_checkpointer_setup' was never awaited
```

`alembic/versions/0001_initial_schema.py:68` hace `asyncio.run(_apply_checkpointer_setup())` pero alembic-async ya tiene un loop corriendo (via `connection.run_sync` desde env.py). Bug heredado de plan 01-03; el comment del migration ("env.py has already returned to sync land by the time upgrade() runs") es incorrecto.

**Pragmatic fix**: `railway ssh --service landa-agent-service "alembic stamp head"` →  marca `alembic_version=0001` sin re-correr el DDL. Es seguro porque las tablas ya las creó `checkpointer.setup()` en el lifespan startup del agent (idempotente). El fix real del migration queda en backlog.

#### Sub-tarea 3.4 — LLM model slug mismatch (commit-less en este SUMMARY; los cambios de defaults se commitearán junto con el SUMMARY)

`POST /test/llm` retornó 500:
```
openai.BadRequestError: Error code: 400 - {'error': {'message': 'google/gemini-2.0-pro is not a valid model ID', ...}}
```

`curl https://openrouter.ai/api/v1/models` confirmó que **no existe `google/gemini-2.0-pro` en OpenRouter**. Gemini 2.0 solo tiene Flash; la línea "pro" arranca en 2.5. CLAUDE.md, settings.py defaults y .env.example asumían 2.0-pro/flash sin verificación contra OpenRouter (deuda heredada del PROJECT.md inicial).

Fix:
- `railway variable set --service landa-agent-service LLM_MODEL_CONVERSATION=google/gemini-2.5-pro LLM_MODEL_JUDGE=google/gemini-2.5-flash LLM_MODEL_INTENT=google/gemini-2.5-flash LLM_MODEL_SUMMARIZER=google/gemini-2.5-flash` (cambio runtime sin redeploy: Railway aplica env vars con un restart, sin rebuild).
- `app/config/settings.py`, `.env.example`, `tests/test_llm_factory.py`, `tests/test_settings.py`, `CLAUDE.md` actualizados a 2.5-{pro,flash} para que defaults locales y prod converjan.

Tras el restart, `POST /test/llm {"text":"di hola en una sola palabra"}` retornó:

  "reply": "Hola",
