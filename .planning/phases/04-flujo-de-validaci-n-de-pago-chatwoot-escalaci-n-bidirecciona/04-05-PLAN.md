    VALID_ACTIONS = frozenset({"aprobar","rechazar","info"})

    def parse_button_id(raw: str) -> tuple[str, str] | None:
        if not raw or "|" not in raw:
            return None
        action, case_id = raw.split("|", 1)
        if action not in VALID_ACTIONS or not case_id:
            return None
        return (action, case_id)


    `async def resume_payment_interrupt(*, action: Literal["aprobar","rechazar","info"], case_id: str, extra: str | None, qa_graph, db_session_factory) -> bool`:
    - Open async session, `select(Case).where(Case.case_id == case_id)`. If None: log warning `cartera.resume.case_not_found`, return False.
    - If `case.status in {"approved","rejected","escalated","closed"}`: log info `cartera.resume.idempotent_skip` with action and existing status; return True (idempotent).
    - `config = {"configurable":{"thread_id": case.phone}}`.
    - `resume_value = {"action": action, "extra": extra}`.
    - `await qa_graph.aupdate_state(config, values={"_cartera_resume": resume_value}, as_node="node_awaiting_cartera")`. (`node_awaiting_cartera` reads `_cartera_resume` from state on resume — adjust Plan 04-04 node if it currently expects the raw interrupt return value; alignment: node reads `state["_cartera_resume"]` after resume.) — coordinate with Plan 04-04 if the field name differs; this plan owns the canonical name `_cartera_resume`.
    - `await qa_graph.ainvoke(None, config=config)`. The graph proceeds from `node_awaiting_cartera` to confirming/escalating per the action.
    - Return True.

    NOTE: LangGraph 1.2.x resume semantics for `interrupt()` may differ slightly — verify the exact resume API in the LangGraph source (`from langgraph.types import Command`). If `Command(resume=value)` is the required pattern instead of `aupdate_state`, use:

    await qa_graph.ainvoke(Command(resume={"action":action,"extra":extra}), config=config)

    Prefer this if available; it's the LangGraph-recommended pattern for resuming interrupts. Either path satisfies the behavior.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Meta webhook → app | All inbound; HMAC verified upstream (F2 invariant); same trust as before |
| Cartera number → app (via Meta) | Trusted by allowlist source-check; tap-only data extraction |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05-01 | Spoofing | Attacker sends message claiming to be cartera | mitigate (D-06) | `CARTERA_PHONE_ALLOWLIST` membership check uses normalized E.164 source; attacker without Meta-validated WhatsApp control over the cartera number cannot spoof source. Meta enforces phone-number authenticity at the Cloud API layer. |
| T-04-05-02 | Tampering | Crafted button_id like `aprobar|<other_case>` to confirm someone else's case | mitigate | `resume_payment_interrupt` re-loads case from DB by `case_id`; resume only proceeds if `case.status="awaiting_cartera"`. Attacker would need both cartera-number access AND a valid open case_id — same authorization as legitimate cartera. Documented in code comment. |
| T-04-05-03 | Repudiation | Double-tap of approve button | mitigate | Idempotency check on `case.status` in `resume_payment_interrupt`. |
| T-04-05-04 | Information disclosure | Reply to unknown number leaks bot existence | mitigate (D-06) | Silent drop — log only, no outbound. Test verifies. |
| T-04-05-05 | Elevation of privilege | Free text from cartera parsed as approval | mitigate (D-05) | Tap-only: `parse_button_id` rejects anything not matching `(aprobar|rechazar|info)|<case_id>`. Free text triggers button re-send, never state change. |
| T-04-05-06 | DoS | Cartera floods text messages → infinite button re-send loop | accept | Rate limiter on `send_buttons` is Phase 5 concern; in v1 the cartera is a known human and unlikely to flood. Documented in known limitations. |
| T-04-05-SC | Tampering | No new package | mitigate | N/A |
</threat_model>
