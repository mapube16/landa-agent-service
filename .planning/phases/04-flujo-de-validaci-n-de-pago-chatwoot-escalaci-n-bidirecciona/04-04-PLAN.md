    `MIME_TO_EXT = {"image/jpeg":"jpg","image/png":"png","image/webp":"webp","application/pdf":"pdf"}`

    `def build_attachment_path(case_id: str, wamid: str, mime: str, now: datetime | None = None) -> Path`:
    - `now = now or datetime.now(timezone.utc)`
    - `ts = now.strftime("%Y%m%dT%H%M%SZ")`
    - `ext = MIME_TO_EXT.get(mime)` — if None raise ValueError.
    - return `Settings().payment.volume_path / case_id / f"{ts}-{wamid}.{ext}"`

    `def store_attachment(data: bytes, case_id: str, wamid: str, declared_mime: str) -> tuple[Path, str]`:
    - `if len(data) > ATTACHMENT_MAX_BYTES: raise ValueError("attachment_too_large")`
    - `if not validate_magic_bytes(data[:16], declared_mime): raise ValueError("magic_byte_mismatch")`
    - `final = build_attachment_path(case_id, wamid, declared_mime)`
    - `final.parent.mkdir(parents=True, exist_ok=True)`
    - `partial = final.with_suffix(final.suffix + ".partial")`
    - `partial.write_bytes(data)`
    - `os.replace(partial, final)`
    - `sha = hashlib.sha256(data).hexdigest()`
    - return `(final, sha)`

    No DB writes here — persistence stays in the node so we can monkeypatch the volume_path in tests without DB.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → app (inbound media) | Untrusted bytes; magic-byte gate + size gate |
| app → Railway volume | Filesystem write inside service container |
| app → Postgres (cases/attachments) | Trusted DB; parameterized queries via SQLAlchemy |
| app → Meta CDN (download) | Outbound bearer-authed |
| app → cartera number | Outbound through Meta; allowlist set in env |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | Tampering | Comprobante mime spoofed | mitigate (D-26) | `validate_magic_bytes` runs before disk write in `store_attachment`. |
| T-04-04-02 | Information disclosure | Comprobante passed to LLM | mitigate (D-27) | Path + bytes are NEVER injected into messages list; only `attachment_count`, `case_id`, `payment_status` reach the LLM via state. Code review enforces. |
| T-04-04-03 | Tampering | Path traversal via wamid containing `..` | mitigate | `build_attachment_path` does not interpolate the wamid into a parent dir; wamid is filename suffix only. Add filename sanitization `wamid = re.sub(r'[^A-Za-z0-9_-]','',wamid)` in `build_attachment_path`. |
| T-04-04-04 | DoS | Disk fill via repeated 5 MB uploads | accept | Plan 04-06 adds 90-day retention cleanup; volume alert at 70% (D-01) catches before exhaustion. |
| T-04-04-05 | Spoofing | LLM hallucinates "pago confirmado" before cartera approval | mitigate (D-28) | Output firewall (wired in Plan 04-05/04-08); `node_confirming` is the ONLY path that sets `payment_approved=True`. |
| T-04-04-06 | Elevation of privilege | Forged interrupt resume | mitigate | Resume can only be triggered by Plan 04-05's cartera-allowlist + button-tap path; no other entry calls `aupdate_state` on payment threads. |
| T-04-04-07 | Repudiation | Same media uploaded twice | mitigate (D-09) | Inbound webhook dedup by `wa:msg:{id}` (F2 invariant); `case_id` reuse logic prevents duplicate case rows. |
| T-04-04-SC | Tampering | npm/pip/cargo installs | mitigate | No new package; all imports already pinned. |
</threat_model>
