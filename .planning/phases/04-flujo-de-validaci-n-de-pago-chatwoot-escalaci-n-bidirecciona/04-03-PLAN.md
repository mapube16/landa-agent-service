<objective>
Build the Chatwoot → client bidirectional channel (D-15, D-16, D-17, D-18). Populate the inverse Redis index when a conversation is created, expose a HMAC-verified webhook that relays human-agent text + media back to the client via Meta Cloud API, and filter out bot-sourced messages to prevent feedback loops.

Purpose: Without this plan, a human agent in Chatwoot cannot reach the client; the escalation path is one-way. This unblocks the success criterion "Agente humano responde desde Chatwoot → mensaje llega al cliente vía WhatsApp".

Output:
- `ChatwootClient.get_or_create_conversation` populates `chatwoot:phone_by_conv:{conv_id}` alongside the existing forward index.
- `ChatwootClient.get_phone_by_conv(conv_id)` Redis-first + API-fallback lookup.
- New router `app/webhooks/chatwoot.py` with HMAC verify + dedup + filter + dispatch.
- Router registered in `app/main.py`.
- Unit tests covering HMAC reject, dedup, agent_bot filter, text relay, media relay.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Chatwoot → app (webhook) | Untrusted POST; HMAC verifies origin |
| app → Meta Graph API | Outbound relays — existing trust boundary, same controls |
| app → Chatwoot API (fallback lookup) | Bearer-authed; standard |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03-01 | Spoofing | Forged Chatwoot webhook | mitigate | HMAC-SHA256 verify against `CHATWOOT_WEBHOOK_SECRET`; `hmac.compare_digest`. |
| T-04-03-02 | Repudiation | Replay of same message | mitigate (D-17) | Redis dedup `chatwoot:msg:{id}` 24h TTL. |
| T-04-03-03 | DoS | Infinite Chatwoot → Meta → mirror → Chatwoot loop | mitigate (D-15) | Filter `sender.type == "user"` (drop `agent_bot`); the mirror posts have `agent_bot` sender, so they're rejected. |
| T-04-03-04 | Information disclosure | Conversation content logged | mitigate | Log only `msg_id`, `conv_id`, hashed phone; content not logged. |
| T-04-03-05 | Tampering | Attachment URL points at attacker server | mitigate | Only download from Chatwoot's own host (validate that `data_url` host matches `settings.chatwoot.base_url` host before download); reject otherwise. |
| T-04-03-06 | Elevation of privilege | Attachment containing executable forwarded | mitigate | Mime allowlist (image/* + application/pdf) before upload to Meta; magic-byte check not applied here because Plan 04-04 enforces it on inbound comprobante path — agent-uploaded files come from Chatwoot, trusted source. Document this asymmetry in the route docstring. |
| T-04-03-SC | Tampering | New package install | mitigate | No new packages (FastAPI, httpx, hmac, mimetypes all already pinned). |
</threat_model>
